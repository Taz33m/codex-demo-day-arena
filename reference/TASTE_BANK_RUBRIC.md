# Taste Bank Rubric

A product is not serious merely because it has code, docs, and tests. A serious product should show product density, workflow specificity, artifact quality, demo compression, repo seriousness, evaluation substance, and anti-toy discipline.

## 1. Product Density

The surface should contain enough workflow, state, decision context, and useful output to feel like a product, not a demo shell. A thin UI around one script is not enough.

## 2. Workflow Specificity

The product should clearly serve a specific operator doing a specific job. The buyer should be able to recognize their workflow without reading a long pitch.

## 3. Artifact Quality

The generated output should be something a real user could review, edit, forward, save, compare, or act on. Logs, toy summaries, and generic reports do not count.

## 4. Demo Compression

The core value should be visible within 60 seconds. A judge should see the painful input, the product action, and the buyer-relevant output without explanation gymnastics.

## 5. Repo Seriousness

The repo should have clear structure, setup, tests/evals, a demo command, a receipt command, and honest limitations. It should look like a product engine with a surface, not a pile of scripts.

## 6. Evaluation Substance

Tests and evals should measure behavior or output quality, not merely file existence. Strong evals compare against baselines, verify customer-relevant fields, and make failure modes visible.

## 7. Anti-Toy Standard

Avoid:

- generic AI dashboards
- thin HTML wrappers
- script-plus-report products
- fake metrics
- vague user personas
- pitch language unsupported by product behavior

The best products in this arena should make the investor feel: this is early, imperfect, and still already pointed at a real company.
