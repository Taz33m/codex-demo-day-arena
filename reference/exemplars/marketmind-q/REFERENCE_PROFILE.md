# Reference Profile: MarketMind-Q

## Why This Is In The Taste Bank

MarketMind-Q is a taste anchor for proof discipline: experiments, baselines, and explicit claims about what the system can and cannot prove. It sets a bar for technical humility and evaluation rigor.

## Product Surface

The bar to borrow is a surface that makes evidence legible. A serious product should let users see scenario inputs, model or engine choices, outputs, comparisons, and limits.

## Workflow Clarity

MarketMind-Q-style quality means the product does not ask the user to trust a black box. It makes assumptions, baselines, and output quality visible enough for a technical buyer or investor to inspect.

## Demo Flow

A strong equivalent demo shows a claim being tested. The product runs a scenario, produces a result, compares it against a baseline or expectation, and exposes the conclusion without overclaiming.

## Repo Quality

Borrow the experimental structure: reproducible commands, scenario fixtures, separated engine code, tests/evals, and clear generated artifacts. The repo should make the proof path easy to rerun.

## Proof / Evaluation Quality

The important standard is measured evidence. Evals should check whether the output is useful for the target workflow, not whether a file exists or a string appears.

## What To Copy

Standards only:

- Baseline comparisons
- Explicit assumptions and limits
- Reproducible evaluation paths
- Scenario-driven proof
- Technical humility around claims

## What Not To Copy

- Do not copy the product idea.
- Do not copy branding.
- Do not copy UI layout blindly.
- Do not reuse code unless explicitly permitted.
- Do not use exotic technical framing as a substitute for buyer usefulness.

## Calibration Questions

1. What would make a new repo feel weaker than this?
2. What does this repo prove quickly?
3. What is the closest equivalent of its aha moment for the current product?
